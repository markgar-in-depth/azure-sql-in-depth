# Chapter 22: Migrating to Azure SQL Database

You've built the app on SQL Server. It works. Now someone decides it's time to move it to Azure SQL Database — and suddenly you're staring at a checklist of migration methods, compatibility gaps, and post-migration tasks that nobody warned you about.

This chapter walks you through the migration paths into Azure SQL Database, the gotchas that will bite you mid-flight, and the operational setup you need once you land. If you're migrating to Managed Instance instead, skip to Chapter 23 — the tooling and trade-offs are different enough to deserve their own chapter.

## Migration Paths for SQL Database

There's no single "right" way to migrate. The best path depends on your database size, your tolerance for downtime, and how many databases you're moving at once. Here are your options.

> **Tip:** Whichever path you choose, scale your target database to a higher service tier during migration to maximize log throughput. Azure SQL Database governs the transaction log rate, and a higher tier means a higher ceiling. Scale back down after the migration completes.

### Azure Database Migration Service

<!-- Source: migrate-from-sql-server/to-azure-sql-database/_summary.md -->

**Azure Database Migration Service (DMS)** is Microsoft's managed migration tool, and it's the path most teams should try first. DMS handles schema migration, data copy, and progress monitoring through a single portal experience.

DMS currently supports **offline** migration to Azure SQL Database. In an offline migration, application downtime begins the moment you start the migration and lasts until the data copy completes. There's no online (continuous sync) mode for Azure SQL Database targets — if you need near-zero downtime, look at transactional replication instead.

Here's what happens under the hood:

1. You create a DMS instance in the Azure portal and install a **self-hosted integration runtime (SHIR)** on a machine that can reach your source SQL Server.
2. You connect to the source, select databases, and map them to target databases in Azure SQL Database.
3. Optionally, you check **Migrate Missing Schema** to push schema objects — tables, views, stored procedures, indexes, functions, and more — before data migration starts.
4. DMS copies data table by table, disabling autostats, triggers, and indexes on the target during the copy, then rebuilds indexes when it finishes.

> **Tip:** DMS copies data table by table — the wider your log throughput ceiling, the faster this goes. Business Critical standard-series caps at 100 MiB/s; Hyperscale premium-series reaches 150 MiB/s.
<!-- Source: azure-sql-database-sql-db/concepts/resource-limits/single-database-resources/resource-limits-vcore-single-databases.md -->

#### DMS Limitations

DMS uses Azure Data Factory pipelines internally, so ADF limits apply:

| Limitation | Detail |
|---|---|
| Tables per database | 100,000 |
<!-- TODO: source needed for "100,000 tables per database DMS limit" -->
| Concurrent migrations | 10,000 per service |
| Double-byte table names | Not supported |
| Computed columns | Not migrated |
| Semicolons in DB names | Not supported |
| BACPAC files over 150 GB | Use SqlPackage CLI |

> **Gotcha:** DMS scales poorly when a database has thousands of tables. Each table incurs ADF activity startup overhead — even a one-row table takes a few seconds to initialize. For table-heavy databases, expect a longer overall migration time than the raw data volume would suggest.

The SHIR machine acts as the compute for the migration. Make sure it has enough CPU and memory for the data copy workload — an undersized SHIR will bottleneck your throughput regardless of how beefy the target database is.

### BACPAC Import

<!-- Source: shared-sql-db-sql-mi-docs/shared-how-tos/load-and-move-data/database-import.md -->

A **BACPAC** file packages both schema and data into a single `.bacpac` archive. You export from the source SQL Server using SqlPackage, SSMS, or PowerShell, then import into Azure SQL Database through the Azure portal, PowerShell, or SqlPackage.

BACPAC is the simplest approach for smaller databases — think line-of-business apps under 150 GB. The entire process is offline: export, upload, import. No continuous sync, no incremental changes.

The Azure portal can import a BACPAC stored in Azure Blob Storage into a new single database. For anything more complex — importing to an existing database, larger files, or automated pipelines — use SqlPackage from the command line.

> **Warning:** BACPAC files over 150 GB generated by SqlPackage may fail to import through the portal or PowerShell with a "corrupted data" error. This is a known issue. Use the SqlPackage CLI directly for large files.

> **Tip:** Minimize the network distance between your BACPAC file and the destination Azure region. Upload the BACPAC to a storage account in the same region as your target server before importing.

For best import performance:

- Disable auto-update and auto-create statistics during import.
- Drop and recreate indexed views after import.
- Partition large tables and indexes ahead of time.

### Transactional Replication

<!-- Source: azure-sql-database-sql-db/how-to/load-and-move-data/replication-to-sql-database.md -->

When you can't afford to take the source database offline for the duration of the migration, **transactional replication** is your answer. It continuously pushes changes from the source SQL Server to Azure SQL Database, letting you run the two systems in parallel until you're ready to cut over.

Azure SQL Database can act as a **push subscriber** in a one-way transactional or snapshot replication topology. It can't be a publisher or distributor — those roles stay on SQL Server.

| Replication type | SQL Database support |
|---|---|
| Standard transactional | Subscriber only |
| Snapshot | Subscriber only |
| Merge | No |
| Peer-to-peer | No |
| Bidirectional | No |

The migration workflow looks like this:

1. Configure distribution on the source SQL Server (or a dedicated distributor).
2. Create a publication on the source database.
3. Create a push subscription pointing to your Azure SQL Database.
4. The snapshot agent seeds the target with an initial copy, then the log reader agent continuously applies changes.
5. Once the target is caught up, redirect your applications and drop the publication.

> **Gotcha:** Only push subscriptions are supported for Azure SQL Database. You can't pull from the subscriber side. The distribution database and replication agents must live on SQL Server, not in Azure SQL Database.

> **Tip:** You can replicate a subset of your database — specific tables, specific rows, specific columns. This is useful for phased migrations where you move workloads one piece at a time.

#### Replication Version Requirements

The publisher/distributor must run SQL Server 2016 or later for full support. Older versions work with specific cumulative updates:

- SQL Server 2014: RTM CU10 or SP1 CU3 minimum
- SQL Server 2012: SP2 CU8 or SP3 minimum

Always use the latest version of SSMS to configure replication to Azure SQL Database.

#### Replication Limitations

Transactional replication to Azure SQL Database has its own set of unsupported operations. If your workload depends on any of these, you'll see query failures or degraded performance until you recreate them on the target:

- File group associations aren't copied
- Table and index partitioning schemes aren't copied
- Fulltext indexes, XML indexes, and spatial indexes aren't copied
- Permissions, extended properties, and data compression attributes aren't copied
- Filtered indexes aren't copied

Plan to script these objects before migration and apply them immediately after the initial schema deployment.

### Other Migration Tools

For completeness, here are additional options that fill specific niches:

| Tool | Best for |
|---|---|
| **BCP (Bulk Copy)** | Data-only moves with full downtime tolerance |
| **Azure Data Factory** | ETL-style migrations, multi-source merges |
| **Azure Migrate** | Discovery and assessment at scale |
| **PowerShell / Azure CLI** | Scripted, repeatable migrations |

BCP and ADF are particularly useful when you need to transform data during the migration or merge data from multiple sources. Azure Migrate shines in the discovery phase — assessing hundreds of SQL Server instances and recommending target SKUs.

### Choosing a Migration Path

| Factor | DMS (Offline) | BACPAC | Tx Replication |
|---|---|---|---|
| Downtime required | Yes | Yes | Minimal |
| Schema migration | Built-in | Built-in | Manual |
| Max practical size | Large | < 150 GB (portal) | Large |
| Continuous sync | No | No | Yes |
| Complexity | Low | Low | Medium–High |
| Subset migration | No | No | Yes |

For most teams, **DMS offline** is the starting point. If downtime is unacceptable, use **transactional replication**. If you're moving a small database and want the simplest possible workflow, use **BACPAC**.

### Custom RBAC Roles for Migration Workflows

<!-- Source: migrate-from-sql-server/to-azure-sql-database/custom-roles.md -->

DMS requires specific Azure permissions to orchestrate migrations. By default, you need Owner or Contributor at the subscription level to provision a DMS instance — but that's more access than most teams should hand out.

Instead, create a custom RBAC role scoped to just the resource groups involved. The role needs actions on `Microsoft.Sql/servers/*` and `Microsoft.DataMigration/*` resource providers. Here's the essential permission set:

```json
{
  "roleName": "SqlDbMigrationOperator",
  "description": "Least-privilege role for DMS migrations to Azure SQL Database",
  "assignableScopes": [
    "/subscriptions/<sub>/resourceGroups/<sql-rg>",
    "/subscriptions/<sub>/resourceGroups/<dms-rg>"
  ],
  "permissions": [{
    "actions": [
      "Microsoft.Sql/servers/read",
      "Microsoft.Sql/servers/write",
      "Microsoft.Sql/servers/databases/read",
      "Microsoft.Sql/servers/databases/write",
      "Microsoft.Sql/servers/databases/delete",
      "Microsoft.DataMigration/databaseMigrations/*",
      "Microsoft.DataMigration/sqlMigrationServices/*",
      "Microsoft.DataMigration/locations/*/read"
    ]
  }]
}
```

Assign this role to the identity running migrations — whether that's a human operator, a service principal, or a managed identity in your CI/CD pipeline. This follows least-privilege without blocking any DMS functionality.

## Post-Migration Operations

<!-- Source: azure-sql-database-sql-db/how-to/migrate/manage-data-after-migrating-to-database.md -->

Landing the data is only half the job. Once your database is in Azure SQL Database, you need to configure the operational envelope that SQL Server handled differently on-premises: monitoring, BCDR, security, and ongoing data movement.

### Monitoring

On-premises, you probably had a monitoring stack built around Perfmon counters, SQL Agent alerts, and maybe a third-party tool. In Azure SQL Database, the platform handles infrastructure monitoring, but you still need application-level visibility.

Your key tools:

- **Azure portal metrics** — CPU percentage, DTU percentage, Data IO percentage, log write percentage, and session counts are all available out of the box. Set up alert rules so you know when the database is running hot before your users do.
- **DMVs** — `sys.dm_db_resource_stats` gives you resource consumption for the last hour at 15-second granularity. `sys.resource_stats` in the `master` database stores 14 days of history.
- **Query Performance Insight** — visualizes top resource-consuming and long-running queries. Requires Query Store to be enabled (it's on by default in Azure SQL Database).
- **Database watcher** — collects in-depth workload monitoring data into a central data store you can query, visualize, and integrate with downstream systems.

> **Tip:** After migration, run `UPDATE STATISTICS` with a full scan on every table. The migration process doesn't carry over statistics, and the query optimizer will produce terrible plans without them.

### Business Continuity and Disaster Recovery

Azure SQL Database handles backups automatically — full backups weekly, differential backups every 12–24 hours, log backups approximately every 10 minutes. You get point-in-time restore (PITR) within the retention period, which varies by service tier.
<!-- Source: azure-sql-database-sql-db/concepts/business-continuity/automated-backups-overview.md -->

But you need to actively configure the rest of your BCDR story:

- **Geo-restore** — available by default. Backups are stored in geo-replicated storage, so you can restore to another region after a regional disaster. Recovery time is measured in hours, not minutes.
- **Active geo-replication** — creates readable secondary replicas in other regions. You manually initiate failover. Up to four secondaries in different regions. <!-- Source: azure-sql-database-sql-db/concepts/business-continuity/disaster-recovery/active-geo-replication-overview.md -->
- **Failover groups** — automate failover across regions for groups of databases. This is the recommended approach for production workloads that need automated regional failover (covered in depth in Chapter 13).
- **Zone redundancy** — distributes replicas across availability zones within a region. Enable this for resilience against datacenter failures.

On-premises, you managed Always On availability groups and failover cluster instances yourself. In Azure SQL Database, high availability is built into the platform — General Purpose uses remote storage with automatic failover, Business Critical uses a local replica architecture similar to Always On AGs. You don't configure any of this; it just works.

### Security Configuration

Your on-premises security model doesn't translate directly. Here's what to set up immediately after migration:

**Authentication.** Windows authentication isn't supported. If you were using AD accounts on-premises, federate your on-premises Active Directory with Microsoft Entra ID, then use Microsoft Entra authentication. SQL logins still work, but Entra is the recommended path for centralized identity management.

> **Important:** Manually recreate SQL logins on the target server. Azure SQL Database doesn't migrate logins automatically — only database users inside the BACPAC or DMS schema migration are carried over. Microsoft provides a [PowerShell utility](https://www.microsoft.com/download/details.aspx?id=103111) that generates T-SQL scripts to recreate logins and map Windows AD accounts to Entra accounts.

**Firewall rules.** By default, all connections are blocked except from other Azure services. Configure server-level firewall rules to allow your application servers and developer machines. Better yet, use virtual network service endpoints or private endpoints to keep traffic off the public internet entirely (see Chapter 6).

**Auditing.** Enable SQL Database auditing to track database events. You can write audit logs to Azure Storage, Log Analytics, or Event Hubs. This replaces SQL Server Audit, which isn't available in SQL Database.

**Threat detection.** Microsoft Defender for SQL detects anomalous database activity — SQL injection attempts, unusual access patterns, brute-force login attacks. Enable it and configure email notifications.

**TDE.** Transparent Data Encryption is enabled by default in Azure SQL Database with Microsoft-managed keys. If you need customer-managed keys, configure Azure Key Vault integration. Unlike on-premises TDE, there's no certificate to migrate — the platform manages it.

**Always Encrypted.** If you're using Always Encrypted on-premises, the column master keys and column encryption keys need to be set up on the target. The encrypted data migrates as-is (it's just ciphertext), but the client-side key configuration must match.

### Data Movement After Migration

Once you're running in Azure SQL Database, you'll still need to move data in and out. The main tools:

- **BACPAC export/import** — export your Azure SQL database to a BACPAC for archival, environment cloning, or moving between subscriptions. Available through the portal, PowerShell, or SqlPackage.
- **BCP** — bulk copy utility for fast data loads. Works the same as on-premises. For large-volume inserts, use the [Smart Bulk Copy tool](https://github.com/Azure-Samples/smartbulkcopy) to parallelize the work.
- **Elastic query** — query across databases without physically moving data. Useful for reporting scenarios where you need to join data from multiple Azure SQL databases.
- **SQL Data Sync** — bidirectional sync between Azure SQL Database and on-premises SQL Server databases. Requires a sync agent installed on-premises.

## T-SQL Compatibility Gaps

<!-- Source: azure-sql-database-sql-db/concepts/transact-sql-tsql-differences-sql-server.md -->

Azure SQL Database doesn't support everything SQL Server does. The differences that bite hardest during migration: **cross-database queries** (three- and four-part names fail), **SQL Server Agent** (replace with elastic jobs or Azure-external scheduling — see Chapter 20), **CLR assemblies**, **linked servers / OPENQUERY**, and **Windows authentication** (use Microsoft Entra instead). Features like FILESTREAM, Service Broker, and server-scoped triggers have no direct equivalent — you'll need Azure-native replacements.

> **Gotcha:** The `USE` statement doesn't work in Azure SQL Database. Each connection is bound to a single database. If your application scripts switch databases with `USE`, every occurrence needs to become a separate connection.

For the full T-SQL compatibility reference — including every unsupported statement, every partial difference, and recommended workarounds — see Chapter 5.

With your database migrated, your security configured, and your compatibility gaps addressed, you're running on Azure SQL Database. The next chapter covers migrating to Managed Instance, where the compatibility surface is broader but the migration tooling is different.
